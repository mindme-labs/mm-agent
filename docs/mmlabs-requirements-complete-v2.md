# MMLabs — Полные требования к прототипу (MVP) v2

## Контекст и решения

**Скоуп прототипа:** Демо + Пре-прод режимы, за 2 дня.
**Аудитория:** Потенциальные клиенты-CEO оптовых компаний.
**Стек:** Next.js (App Router) + Payload CMS 3.0 + MongoDB.
**Деплой:** Vercel.

**Ключевое отличие режимов:**
- **Демо** — без AI, фиксированные демо-файлы, захардкоженные рекомендации.
- **Пре-прод** — с AI (Claude API): AI распознаёт формат загруженных файлов, извлекает данные, генерирует тексты рекомендаций и проводит экспертный аудит оборотных средств.

---

## 1. Структура сервиса

Сервис состоит из следующих компонент:

1. **Лендинг** — будет описан отдельно и позже.
2. **Личный кабинет пользователя** — основной рабочий интерфейс CEO (раздел 3).
3. **Административная панель сервиса** — Payload CMS Admin UI на `/admin` (раздел 4).
4. **Сервис анализа файлов** — парсер ОСВ + AI-извлечение данных + rules engine (раздел 5).
5. **Интеграционный сервис для взаимодействия с 1С Бухгалтерией** — заглушка в прототипе (раздел 6).
6. **Интеграционный сервис для взаимодействия с поставщиком AI функций** — **реализуется в пре-прод режиме** (раздел 7).
7. **Сервис биллинга** — не реализуется в прототипе, рекомендация по решению в разделе 8.
8. **Система логирования и аналитики** — сбор всех действий пользователей и данных (раздел 9).

## 2. Режимы работы и процессы (уточнение)

### 2.1. Режимы для прототипа

| Параметр | Демо | Пре-прод |
|----------|------|----------|
| Данные | Предзагруженные демо-CSV из 1С | Пользователь загружает свои CSV/Excel |
| Распознавание файлов | Не требуется (файлы фиксированы) | **AI определяет тип и структуру файлов** |
| Извлечение данных | Захардкоженный парсер для демо-файлов | **AI извлекает данные из файлов произвольного формата** |
| Rules engine | Да, типизированные правила | Да + **AI-аудитор для нетипизированных проблем** |
| Генерация текстов | Шаблоны с подстановкой | **AI генерирует персонализированные тексты** |
| Подключение к 1С | Имитация (анимация) | Нет. Ручная загрузка файлов |
| Сохранение данных | Сессия стирается при выходе | Данные и результаты сохраняются в БД |
| Обратная связь | Доступна (но не сохраняется) | Сохраняется в БД |

### 2.2. Поддерживаемые файлы из 1С

**Для демо-режима** — строго фиксированный формат ОСВ из 1С:Бухгалтерии:

| Счёт | Название | Что даёт для анализа |
|------|----------|---------------------|
| 10 | Материалы | Запасы сырья, оборачиваемость материалов |
| 41 | Товары | Складские запасы, выявление неликвида |
| 45 | Товары отгруженные | Отгрузки без оплаты, риски по контрагентам |
| 60 | Расчёты с поставщиками | Кредиторская задолженность, авансы поставщикам |
| 62 | Расчёты с покупателями | Дебиторская задолженность по контрагентам |
| 90.01 | Выручка | Выручка помесячно по номенклатурным группам |
| 90.02 | Себестоимость продаж | Себестоимость помесячно, расчёт маржи |

**Для пре-прод режима** — файлы могут отличаться по формату. AI-сервис должен:
1. Принять CSV или Excel файл.
2. Определить, какой это документ (ОСВ, карточка счёта, акт сверки и т.п.) и по какому счёту.
3. Извлечь структурированные данные вне зависимости от конкретного форматирования выгрузки.
4. Сообщить пользователю, если файл не удалось распознать.

Подробнее об AI-парсинге — в разделе 5.3.

## 3. Личный кабинет пользователя

### 3.1. Структура экранов

Личный кабинет — это SPA-приложение на маршруте `/app/*` с нижней навигационной панелью.

**Навигация (Bottom Nav):**

| Таб | Иконка | Маршрут | Описание |
|-----|--------|---------|----------|
| Входящие | Inbox/Bell | `/app/inbox` | Финансовая сводка + карточки рекомендаций |
| Мои задачи | CheckSquare | `/app/tasks` | Управление статусами рекомендаций |
| Данные | FileSpreadsheet | `/app/data` | Загруженные файлы + ключевые метрики |

### 3.2. Онбординг (`/app/onboarding`)

Пошаговый визард, отображается при `hasCompletedOnboarding === false`.

**Демо-режим:**
1. **Приветствие** — краткое описание: "Сейчас мы покажем, как сервис находит проблемы в ваших финансах и предлагает решения."
2. **Демо-файлы** — экран со списком предзагруженных ОСВ (7 файлов). Пользователь видит названия файлов и может нажать "Начать анализ".
3. **Анализ** — анимация прогресса (3–5 секунд): "Загрузка данных из 1С → Анализ дебиторской задолженности → Проверка складских остатков → Формирование рекомендаций".
4. **Готово** — "Диагностика завершена. Найдено N проблем, требующих внимания." Кнопка "Перейти к результатам".

**Пре-прод режим:**
1. **Приветствие** — "Загрузите выгрузки из вашей 1С:Бухгалтерии для анализа. Мы принимаем оборотно-сальдовые ведомости и другие отчёты в формате CSV или Excel."
2. **Загрузка файлов** — Drag-n-drop зона + кнопка выбора файлов. Ограничения: до 10 файлов, суммарно до 10 Мб. AI автоматически определяет тип каждого файла и показывает результат распознавания в таблице (тип документа, счёт, период, статус). Если файл не распознан — предупреждение с возможностью пропустить.
3. **Анализ** — реальный парсинг на сервере через AI. Прогресс-бар: "Распознавание файлов → Извлечение данных → Расчёт метрик → Проверка типовых правил → AI-аудит оборотных средств → Формирование рекомендаций".
4. **Рекомендации по ведению учёта** — если AI обнаружил проблемы в структуре данных, показываем советы.
5. **Готово** — аналогично демо.

После завершения онбординга: `hasCompletedOnboarding = true`, редирект на `/app/inbox`.

### 3.3. Экран "Входящие" (`/app/inbox`)

Основной экран, который CEO видит каждый день.

**Верхняя секция — Финансовая сводка:**

| Метрика | Источник | Формат |
|---------|----------|--------|
| Выручка за период | Счёт 90.01, сумма кредитовых оборотов | `₽ X.X млн` |
| Дебиторская задолженность | Счёт 62, дебетовое сальдо на конец периода | `₽ X.X млн` |
| Кредиторская задолженность | Счёт 60, кредитовое сальдо на конец периода | `₽ X.X млн` |
| Рекомендаций к действию | Количество карточек в статусе "new" | число |

Дополнительные метрики (раскрываются по тапу "Подробнее"):

| Метрика | Источник | Формат |
|---------|----------|--------|
| Оборачиваемость ДЗ | Расчёт из сч. 62 + 90.01 | `X дн.` |
| Оборачиваемость КЗ | Расчёт из сч. 60 + 90.02 | `X дн.` |
| Валовая рентабельность | (90.01 − 90.02) / 90.01 | `X%` |

Индикатор здоровья (Health Index) по соотношению ДЗ/КЗ:
- 🟢 **В норме** — ДЗ/КЗ > 1.2
- 🟡 **Есть вопросы** — ДЗ/КЗ от 0.8 до 1.2
- 🔴 **Риск** — ДЗ/КЗ < 0.8

Подпись: "Данные за [период из файлов]".

**Нижняя секция — Лента рекомендаций:**

Карточки проблем в статусе `new`, отсортированы по приоритету (`critical` → `high` → `medium` → `low`).

**Структура карточки рекомендации:**

```
┌────────────────────────────────────────┐
│ 🔴 Критично    ДЗ-2                   │ ← приоритет + код правила
│ Критическая концентрация ДЗ            │ ← формальное название
│                                        │
│ ООО «Вектор» — 38% дебиторки          │ ← заголовок
│                                        │
│ ↓ Дебиторская задолженность −4.5 млн ₽│ ← impact tag
│                                        │
│ Долг контрагента достиг 4 500 000 руб. │
│ Это 38% от всей ДЗ. Задержка их      │
│ платежа парализует бизнес.             │ ← описание
│                                        │
│ 💡 Рекомендация:                       │
│ Предложить скидку 1.5% за досрочное   │
│ погашение части долга.                 │ ← краткая рекомендация
│                                        │
│ [Взять в работу]  [Отклонить]          │ ← действия
│                                        │
│ 📋 Скопировать текст для отправки      │ ← копирование
│                                        │
│ ─────────────────────────────────────  │
│ Эта рекомендация полезна?              │
│ [👍 Да]  [👎 Нет]  [💬 Комментарий]   │ ← обратная связь
└────────────────────────────────────────┘
```

**Действия с карточкой:**

| Действие | Кнопка | Новый статус | Описание |
|----------|--------|-------------|----------|
| Взять в работу | Primary, синяя | `in_progress` | CEO передаёт задачу ответственному лицу |
| Отклонить | Secondary, серая | `dismissed` | CEO считает рекомендацию нерелевантной |
| Скопировать текст | Иконка + текст, отдельная строка | — | Копирует полный текст рекомендуемого сообщения в буфер |

**Обратная связь по рекомендации:**

Под каждой карточкой — компактная секция "Эта рекомендация полезна?" с тремя элементами:
- **👍 Да** — одно нажатие, фиксирует положительную оценку.
- **👎 Нет** — одно нажатие, фиксирует отрицательную оценку.
- **💬 Комментарий** — раскрывает текстовое поле для свободного комментария (макс. 500 символов). Кнопка "Отправить".

После нажатия 👍/👎 кнопки заменяются на "Спасибо за отзыв". Обратная связь сохраняется в коллекции `RecommendationFeedback` (раздел 4.1) и логируется (раздел 9).

Обратная связь работает в обоих режимах, но в демо-режиме данные не сохраняются в БД (только отображается UI).

### 3.4. Экран "Мои задачи" (`/app/tasks`)

Все рекомендации, сгруппированные по статусам.

**Табы:**

| Таб | Статус | Бейдж |
|-----|--------|-------|
| В работе | `in_progress` | Количество |
| Решены | `resolved` | Количество |
| Отклонены | `dismissed` | — |

Внутри каждого таба — список карточек. На каждой карточке — dropdown для смены статуса на любой другой. Карточки в этом экране также содержат секцию обратной связи.

### 3.5. Экран "Данные" (`/app/data`)

Информация о загруженных данных и ключевых метриках.

**Секция 1 — Загруженные файлы:**
Таблица с файлами: название, распознанный тип, счёт, период, дата загрузки, статус (✅ / ⚠️).
В пре-прод режиме: кнопка "Загрузить новые файлы" для повторного анализа.

**Секция 2 — Ключевые метрики (расширенные):**

| Метрика | Расчёт |
|---------|--------|
| Выручка за период | Счёт 90.01, сумма кредитовых оборотов за все месяцы |
| Себестоимость за период | Счёт 90.02, сумма дебетовых оборотов |
| Валовая прибыль | Выручка − Себестоимость |
| Рентабельность | Валовая прибыль / Выручка × 100% |
| ДЗ на конец периода | Счёт 62, итоговое дебетовое сальдо |
| КЗ на конец периода | Счёт 60, итоговое кредитовое сальдо |
| Запасы на конец периода | Счёт 41 + Счёт 10, итоговое дебетовое сальдо |
| Товары отгруженные | Счёт 45, итоговое дебетовое сальдо |

**Секция 3 — Топ-5 дебиторов:**
Таблица из данных счёта 62: контрагент, сумма долга, доля от общей ДЗ.

**Секция 4 — Топ-5 кредиторов:**
Таблица из данных счёта 60: контрагент, сумма долга, наличие авансов.

## 5. Сервис анализа файлов с выгрузкой по счетам

### 5.1. Два режима работы парсера

**Демо-режим — детерминированный парсер:**
Файлы фиксированы, формат известен заранее. Используется жёсткий парсер, описанный в 5.2.

**Пре-прод режим — AI-парсер:**
Формат файлов может отличаться от стандартного. Вместо жёсткого парсера используется двухэтапный AI-процесс:
1. **AI-распознавание** — определение типа документа и структуры (раздел 5.3).
2. **AI-извлечение данных** — преобразование в стандартную структуру (раздел 5.3).

После извлечения данные попадают в единый rules engine (раздел 5.4) + AI-аудит (раздел 5.5).

### 5.2. Детерминированный парсер (демо-режим)

Все файлы — стандартные ОСВ из 1С:Бухгалтерии в CSV.

**Общая структура CSV:**
```
Строка 1: "Оборотно-сальдовая ведомость по счету XX за YYYY г."
Строка 2: пустая
Строка 3: заголовки (зависят от счёта)
Строка 4: подзаголовки колонок
Строка 5: пустая или "Период"
Строки 6+: данные
Последняя строка: "Итого"
```

**Разделитель:** `;`. **Десятичный разделитель:** `,`. **Разделитель тысяч:** пробел. **Кодировка:** UTF-8 с BOM.

**Структуры данных по типам счетов:**

Счета 10, 41 (Материалы, Товары) — иерархия: Номенклатура → Месяц (8 колонок, включая "Показатели" и "Кол.").

Счёт 45 (Товары отгруженные) — иерархия: Контрагент → Месяц.

Счета 60, 62 (Поставщики, Покупатели) — иерархия: Контрагент → [Договор →] Месяц (7 колонок).

Счета 90.01, 90.02 (Выручка, Себестоимость) — иерархия: Номенклатурная группа → Месяц.

**Алгоритм:**
1. Идентификация: извлечь номер счёта и период из строки 1.
2. Определение структуры: найти заголовки, определить маппинг колонок.
3. Парсинг: построчно определить тип строки (итого / сущность / месяц / количество / договор).
4. Числа: удалить пробелы, заменить `,` на `.`, привести к float.

**Выходная структура (единая для обоих режимов):**
```typescript
interface ParsedAccountData {
  accountCode: string;
  period: string;
  totals: {
    openingDebit: number;
    openingCredit: number;
    turnoverDebit: number;
    turnoverCredit: number;
    closingDebit: number;
    closingCredit: number;
  };
  entities: Array<{
    name: string;
    totals: { ... };
    monthly: Array<{
      month: string;
      turnoverDebit: number;
      turnoverCredit: number;
      closingDebit: number;
      closingCredit: number;
    }>;
  }>;
}
```

### 5.3. AI-парсер (пре-прод режим)

Когда пользователь загружает файл в пре-прод режиме, вместо жёсткого парсера запускается двухэтапный AI-процесс.

**Этап 1 — Распознавание файла (промпт `file_recognition`):**

Вход: первые 50 строк файла (текстовое содержимое).
Задача AI: определить тип документа, номер счёта, период, структуру колонок.
Выход: JSON с метаданными.

```
System: Ты — специалист по бухгалтерскому учёту в 1С. Тебе присылают
фрагмент файла с бухгалтерской выгрузкой. Определи:
1. Тип документа (ОСВ, карточка счёта, анализ счёта, акт сверки, иное)
2. Номер бухгалтерского счёта (10, 41, 45, 51, 60, 62, 90.01, 90.02 и т.д.)
3. Период данных
4. Структуру колонок (какая колонка содержит какие данные)
5. Разделитель (точка с запятой, запятая, табуляция)
6. Иерархию данных (по контрагентам, по номенклатуре, по месяцам)

Верни ТОЛЬКО JSON без пояснений:
{
  "documentType": "ОСВ",
  "accountCode": "62",
  "period": "2025",
  "separator": ";",
  "columns": {"0": "account_or_entity", "1": "opening_debit", ...},
  "hierarchy": ["counterparty", "month"],
  "headerRows": 5,
  "confidence": 0.95
}

Если файл не является бухгалтерской выгрузкой — верни
{"documentType": "unknown", "confidence": 0}.
```

**Этап 2 — Извлечение данных (промпт `data_extraction`):**

Вход: полное содержимое файла + результат распознавания с этапа 1.
Задача AI: извлечь структурированные данные в формат `ParsedAccountData`.
Выход: JSON в стандартной структуре.

```
System: Ты — бухгалтер-аналитик. Из предоставленного файла извлеки данные
в строго заданную JSON-структуру. Файл уже распознан как: {recognition_result}.

Извлеки ВСЕ строки данных. Для каждой сущности (контрагента/номенклатуры)
собери итоговые данные и помесячную разбивку. Числа приводи к float
(убери пробелы, замени запятую на точку). Пустые ячейки = 0.

Верни ТОЛЬКО JSON в формате ParsedAccountData (структура предоставлена).
```

**Обработка ошибок:**
- Если `confidence` < 0.7 — файл помечается как `warning`, пользователю показывается: "Не удалось уверенно распознать файл. Пожалуйста, убедитесь, что это выгрузка из 1С."
- Если `documentType === "unknown"` — файл помечается как `error`, пропускается.
- Все AI-вызовы логируются в `AIUsageLogs` и `UploadedFiles.aiRecognitionLog`.

### 5.4. Rules Engine — типизированные правила

После парсинга (любым из двух способов) rules engine проверяет данные по набору формальных правил. Всего 9 правил: 8 бизнес-правил + 1 сервисное.

#### ДЗ-1: Просроченная дебиторская задолженность (счёт 62)

Двухступенчатая логика:

| Ступень | Условие | Приоритет | Действие |
|---------|---------|-----------|----------|
| Мягкая | Дебетовое сальдо > 0, нет оплат 1 мес. | `medium` | Письмо с просьбой о сверке расчётов |
| Жёсткая | Дебетовое сальдо > 0, нет оплат 2+ мес. | `high` (> 500К ₽) / `medium` | Досудебная претензия с расчётом пени |

- Источник: счёт 62, помесячные кредитовые обороты.
- Impact: `accounts_receivable`, `decrease`, сумма сальдо.
- Адресат: бухгалтер / юрист.

#### ДЗ-2: Критическая концентрация дебиторской задолженности (счёт 62)

- Условие: Дебетовое сальдо контрагента > 30% от итога по счёту 62.
- Приоритет: `critical`.
- Impact: `accounts_receivable`, `decrease`, сумма долга.
- Действие: оффер скидки 1–2% за досрочное погашение; альтернатива — факторинг.
- Адресат: CEO.

#### ДЗ-3: Снижение активности ключевых покупателей (счёт 62)

- Условие: Контрагент имел стабильные обороты в 4+ месяцах из первых 6 И нулевые дебетовые обороты (новые отгрузки) в последние 2 месяца.
- Приоритет: `high` (среднемесячный оборот > 200К ₽) / `medium`.
- Impact: `revenue`, `decrease`, среднемесячный оборот × 3.
- Действие: задача менеджеру связаться с клиентом.
- Адресат: менеджер продаж.

#### КЗ-1: Незакрытые авансы поставщикам (счёт 60)

- Условие: Дебетовое сальдо (аванс) И нулевые кредитовые обороты (поступления) 2+ мес.
- Приоритет: `medium` (> 500К ₽) / `low`.
- Impact: `accounts_payable`, `decrease`, сумма аванса.
- Действие: запрос статуса отгрузки.
- Адресат: менеджер закупок.

#### ЗАП-1: Неликвидные складские запасы (счета 41, 10)

- Условие: Дебетовое сальдо > 0, нулевые кредитовые обороты 3+ мес.
- Приоритет: `medium` (> 100К ₽) / `low`.
- Impact: `inventory`, `decrease`, сумма остатка.
- Действие: спец-оффер со скидкой 15%.
- Адресат: менеджер продаж.

#### ЗАП-2: Избыточные складские запасы (счёт 41)

- Условие: Остаток / среднемесячное списание > 180 дней. Позиция НЕ является неликвидом (есть продажи).
- Приоритет: `medium`.
- Impact: `inventory`, `decrease`, сумма превышения над 3-месячным нормативом.
- Действие: рекомендация остановить закупку на квартал.
- Адресат: менеджер закупок.

#### ПЛ-1: Снижение валовой рентабельности (счета 90.01 + 90.02)

- Условие: (Выручка − Себестоимость) / Выручка < 15% за любой месяц.
- Приоритет: `high`.
- Impact: `strategic`.
- Действие: рекомендация пересмотреть ценовую политику.
- Адресат: коммерческий директор / CEO.

#### ФЦ-1: Дисбаланс платёжных циклов (счета 60 + 62)

- Условие: Оборачиваемость ДЗ (дни) > Оборачиваемость КЗ (дни) × 1.5.
- Расчёт оборачиваемости ДЗ: (Среднее сальдо сч. 62 / Обороты Кт сч. 90.01) × дни периода.
- Расчёт оборачиваемости КЗ: (Среднее сальдо сч. 60 / Обороты Дт сч. 90.02) × дни периода.
- Приоритет: `high` (разница > 30 дней) / `medium`.
- Impact: `strategic`.
- Действие: стратегическая рекомендация по пересмотру условий договоров.
- Адресат: CEO / фин. директор.

#### СВС-1: Рекомендации по качеству учётных данных (сервисное)

- Условие: Не загружен файл по важному счёту, или структура не соответствует ожиданиям.
- Приоритет: `low`.
- Примеры: "Для полноценного анализа загрузите ОСВ по счёту 62."

#### Сводная таблица правил

| Код | Формальное название | Счёт | Тип проблемы | Адресат |
|-----|-------------------|------|-------------|---------|
| ДЗ-1 | Просроченная дебиторская задолженность | 62 | Риск невозврата | Бухгалтер / юрист |
| ДЗ-2 | Критическая концентрация ДЗ | 62 | Концентрация риска | CEO |
| ДЗ-3 | Снижение активности ключевых покупателей | 62 | Упущенная выручка | Менеджер продаж |
| КЗ-1 | Незакрытые авансы поставщикам | 60 | Потерянные деньги | Менеджер закупок |
| ЗАП-1 | Неликвидные складские запасы | 41, 10 | Замороженный капитал | Менеджер продаж |
| ЗАП-2 | Избыточные складские запасы | 41 | Затоваривание | Менеджер закупок |
| ПЛ-1 | Снижение валовой рентабельности | 90.01, 90.02 | Эрозия прибыли | Ком. директор |
| ФЦ-1 | Дисбаланс платёжных циклов | 60, 62 | Финансирование рынка | CEO / фин. директор |
| СВС-1 | Качество учётных данных | Все | Качество данных | Бухгалтер |

### 5.5. AI-аудит оборотных средств (пре-прод режим)

В дополнение к типизированным правилам (раздел 5.4), в пре-прод режиме запускается **AI-аудит** — экспертный анализ данных моделью Claude, работающей в роли опытного финансового аудитора. Цель — выявить нетипизированные проблемы, которые не покрываются фиксированными правилами.

**Промпт AI-аудитора (`audit_working_capital`):**

```
System:
Ты — старший финансовый аудитор с 15-летним опытом работы с оптовыми
торговыми компаниями в России. Специализация: оборотный капитал, ликвидность,
операционная эффективность B2B-дистрибуции.

Твоя задача — провести глубокий аудит оборотных средств компании на основе
предоставленных данных из бухгалтерского учёта (оборотно-сальдовые ведомости).

КОНТЕКСТ:
Компания — оптовая торговая компания. Данные из 1С:Бухгалтерии за период {period}.

ДАННЫЕ:
{structured_data_json}

Типизированные правила уже проверили стандартные ситуации (просроченная ДЗ,
неликвид, концентрация долга, забытые авансы, маржинальность, отток клиентов,
оверсток, дисбаланс отсрочек). Тебе НЕ нужно дублировать их.

ПРОВЕДИ АУДИТ ПО СЛЕДУЮЩИМ НАПРАВЛЕНИЯМ:

1. АНОМАЛИИ В ДИНАМИКЕ:
   - Сначала проведи типизацию взаимоотношений с клиентом, с учётом
     сезонности (расчёт стандартного «Y»). Затем анализируй, считая
     отклонения текущего значения «Y» от стандартного значения «Y».
     Y стандартный = ДЗ стандартная на начало периода
                   + Сумма отгрузок стандартная (начислений)
                   − Сумма оплат стандартная
     Y текущий = аналогично по фактическим данным
   - Резкие скачки в помесячных оборотах по контрагентам (рост/падение > 50%)
   - Нетипичные сезонные паттерны
   - Контрагенты с "пилообразной" динамикой (признак нестабильности)

2. СТРУКТУРНЫЕ РИСКИ:
   - Несоразмерность масштабов ДЗ и КЗ общему обороту — смотри в динамике,
     исходи из отклонений от обычной деятельности
   - Аномальное соотношение запасов к объёму продаж — смотри в динамике,
     исходи из отклонений от обычной деятельности
   - Признаки того, что компания растёт быстрее, чем позволяет оборотный капитал

3. СКРЫТЫЕ ВОЗМОЖНОСТИ:
   - Контрагенты (дебиторы и кредиторы) с растущей динамикой закупок
     (потенциал для переговоров о лучших условиях)
   - Номенклатурные группы с растущей маржинальностью
   - Возможности для ускорения оборачиваемости ДЗ и КЗ

4. ПРЕДУПРЕЖДЕНИЯ:
   - Признаки приближающегося кассового разрыва (даже если данных о
     банковском счёте нет — косвенные признаки из ДЗ/КЗ, исходя из
     изменений поведения контрагентов)
   - Контрагенты, чьё поведение изменилось (раньше платили вовремя,
     начали задерживать)
   - Зависимость от одного поставщика (концентрация КЗ)

ФОРМАТ ОТВЕТА:
Верни JSON-массив найденных проблем/возможностей. Только те, которые
ты считаешь значимыми (не более 5). Каждая:
[
  {
    "title": "Краткий заголовок проблемы",
    "description": "Описание с конкретными цифрами из данных",
    "recommendation": "Что конкретно делать",
    "fullText": "Полный текст сообщения ответственному лицу",
    "priority": "critical|high|medium",
    "recipient": "Кому адресовано (CEO, бухгалтер, менеджер...)",
    "impactMetric": "accounts_receivable|accounts_payable|inventory|revenue|strategic",
    "impactDirection": "decrease|increase",
    "impactAmount": 0
  }
]

Если ничего значимого не найдено — верни пустой массив [].
Не придумывай проблемы, которых нет в данных.
```

**Карточки от AI-аудита** создаются с `ruleCode: "AI-AUDIT"` и `isAiGenerated: true`, чтобы отличать их от типизированных правил. В интерфейсе они помечаются бейджем "🤖 AI-аудит".

### 5.6. Шаблоны текстов рекомендаций

Для **демо-режима** — каждый тип правила имеет шаблон `fullText` с переменными в `{}`.
Для **пре-прод режима** — тексты генерируются AI на основе промпта `recommendation_text` (раздел 7.3).

**Шаблон ДЗ-1а — Просроченная ДЗ (мягкая ступень):**
```

Просим провести сверку расчётов по нашему договору поставки.
По нашим данным, за {counterparty} числится задолженность в размере
{amount} руб. Последняя оплата поступала более {months} мес. назад.

Просим подтвердить сумму и при наличии расхождений направить нам
ваш вариант акта сверки.

С уважением,
{companyName}
```

**Шаблон ДЗ-1б — Просроченная ДЗ (жёсткая ступень, досудебная претензия):**
```

{counterparty}

Настоящим уведомляем о просроченной задолженности по договору поставки.

Сумма основного долга: {amount} руб.
Период просрочки: более {months} мес.
Расчёт пени (0.1% в день): {penaltyAmount} руб.
Итого к оплате: {totalAmount} руб.

Просим в срок до {deadline} произвести оплату в полном объёме.

С уважением,
{companyName}
```

## 7. Интеграционный сервис для взаимодействия с поставщиком AI функций

### 7.1. Статус в прототипе

**Реализуется для пре-прод режима.** AI используется в четырёх сценариях:

| Сценарий | Промпт | Когда вызывается |
|----------|--------|------------------|
| Распознавание файлов | `file_recognition` | При загрузке файла в пре-проде |
| Извлечение данных | `data_extraction` | После распознавания файла |
| Генерация текстов рекомендаций | `recommendation_text` | При создании карточки правилами |
| Экспертный аудит оборотных средств | `audit_working_capital` | После обработки всех файлов |

В **демо-режиме** AI не вызывается — используются захардкоженные шаблоны.

### 7.2. Поставщик и модель

**Поставщик:** Anthropic Claude API.
**Модель:** `claude-sonnet-4-20250514` (оптимальный баланс качества и стоимости).
**Библиотека:** `@anthropic-ai/sdk` (npm).

### 7.3. Промпты

Все промпты хранятся в коллекции `AIPrompts` и редактируемы через админку без деплоя.

**Промпт `file_recognition`** — описан в разделе 5.3.
**Промпт `data_extraction`** — описан в разделе 5.3.
**Промпт `audit_working_capital`** — описан в разделе 5.5.

**Промпт `recommendation_text` — генерация текстов рекомендаций:**

```
System:
Ты — бизнес-аналитик, составляющий деловые документы для B2B-компаний.
Пиши на русском языке, в деловом стиле. Текст должен быть готов к
отправке без редактирования.

User:
Тип проблемы: {ruleCode} — {ruleName}
Данные: {structured_data}
Контрагент: {counterparty}
Сумма: {amount}
Адресат: {recipient}

Сгенерируй два текста:
1. shortRecommendation — 1-2 предложения для карточки (до 200 символов)
2. fullText — полный текст сообщения для отправки адресату

Верни JSON: {"shortRecommendation": "...", "fullText": "..."}
```

### 7.4. Учёт токенов

Каждый вызов AI логируется в `AIUsageLogs`: user_id, промпт, input/output tokens, модель, стоимость, время выполнения.

**Примерная стоимость на одного пре-прод пользователя (10 файлов):**
- Распознавание: 10 вызовов × ~1K tokens = ~10K tokens
- Извлечение: 10 вызовов × ~5K tokens = ~50K tokens
- Тексты рекомендаций: ~10 вызовов × ~2K tokens = ~20K tokens
- AI-аудит: 1 вызов × ~10K tokens = ~10K tokens
- **Итого: ~90K tokens ≈ $0.30–0.50 за одного пользователя**

### 7.5. Fallback-стратегия

Если AI недоступен или произошла ошибка:
- Распознавание файлов: попытка использовать детерминированный парсер.
- Генерация текстов: подстановка шаблонов (как в демо-режиме).
- AI-аудит: пропускается, пользователь получает только типизированные рекомендации.
- В EventLog фиксируется событие `ai_fallback`.

## 8. Сервис биллинга

### 8.1. Статус в прототипе

**Не реализуется.** Все пользователи прототипа работают бесплатно.

### 8.2. Модель тарификации (целевая)

**Подписка + потребление:**
- Базовая подписка: фиксированная плата за доступ.
- Потребление: AI-вызовы сверх лимита.

**Тарифные планы (рабочая гипотеза):**

| План | Цена/мес | Файлов/мес | AI-запросов/мес | Режим |
|------|----------|-----------|-----------------|-------|
| Демо | Бесплатно | Демо-данные | 0 | demo |
| Стартовый | 4 990 ₽ | 10 загрузок | 50 | preprod |
| Бизнес | 14 990 ₽ | Безлимит (1С sync) | 500 | production |

### 8.3. Рекомендация: Lago

Для гибкого управления тарифами, начислениями и usage billing рекомендуется **Lago** — open-source billing platform.

**Почему Lago:**

| Критерий | Lago | ЮKassa | Stripe | CloudPayments |
|----------|------|--------|--------|---------------|
| Гибкие тарифы (подписка + usage) | ✅ Нативно | ❌ Нет | ✅ Да | ❌ Нет |
| Управление планами через API/UI | ✅ Dashboard + API | ❌ Только API | ✅ Да | ⚠️ Ограниченно |
| Usage metering (подсчёт AI-токенов) | ✅ Events API | ❌ Нет | ✅ Metered billing | ❌ Нет |
| Счета и инвойсы | ✅ Автогенерация | ❌ Нет | ✅ Да | ❌ Нет |
| Приём платежей (РФ) | ⚠️ Нужен шлюз | ✅ Нативно | ❌ Не РФ | ✅ Нативно |
| Self-hosted (контроль данных) | ✅ Docker | ❌ | ❌ | ❌ |
| Open-source | ✅ AGPLv3 | ❌ | ❌ | ❌ |

**Рекомендуемая архитектура:**

```
Lago (биллинг-ядро)          CloudPayments / ЮKassa (платёжный шлюз)
├── Управление тарифами       ├── Приём платежей в РФ
├── Usage metering            ├── Рекуррентные списания
│   (AI-токены, файлы)        └── 3D-Secure
├── Инвойсы
└── Webhook → наш бэкенд
         ↓
    Обновление mode
    (demo → preprod → production)
```

**Альтернативы:**
- **Stigg** — SaaS для feature flags + billing, не self-hosted, нет РФ-шлюзов.
- **Stripe + ЮKassa** — Stripe для биллинг-логики + ЮKassa как шлюз, но сложная интеграция.
- **Kill Bill** — open-source, мощнее Lago, но значительно сложнее в развёртывании.

## 9. Система логирования и аналитики

### 9.1. Принцип

Все значимые действия пользователей, все передаваемые данные и все полученные рекомендации логируются в коллекцию `EventLog`. Это необходимо для:
- Понимания, как пользователи взаимодействуют с рекомендациями.
- Анализа качества рекомендаций (совместно с RecommendationFeedback).
- Аудита безопасности (кто загрузил какие данные).
- Метрик для продукта (воронка, retention, конверсия в действия).

### 9.2. Типы событий

**Аутентификация:**

| eventType | Когда | payload |
|-----------|-------|---------|
| `auth.login` | Успешный вход | `{mode, isFirstLogin}` |
| `auth.login_denied` | Email не в allowlist | `{email}` |
| `auth.logout` | Выход | — |

**Онбординг:**

| eventType | Когда | payload |
|-----------|-------|---------|
| `onboarding.start` | Начал онбординг | `{mode}` |
| `onboarding.file_upload` | Загрузил файлы (пре-прод) | `{fileCount, totalSize}` |
| `onboarding.analysis_start` | Запустил анализ | — |
| `onboarding.analysis_complete` | Анализ завершён | `{recommendationCount, duration}` |
| `onboarding.complete` | Завершил онбординг | `{mode}` |

**Файлы:**

| eventType | Когда | payload |
|-----------|-------|---------|
| `file.upload` | Загружен файл | `{fileId, originalName, size, mimeType}` |
| `file.recognized` | AI распознал файл | `{fileId, detectedType, accountCode, confidence}` |
| `file.parsed` | Файл распарсен | `{fileId, status, entityCount}` |
| `file.parse_error` | Ошибка парсинга | `{fileId, error}` |

**Рекомендации:**

| eventType | Когда | payload |
|-----------|-------|---------|
| `recommendation.created` | Создана рекомендация | `{recId, ruleCode, priority, isAiGenerated}` |
| `recommendation.viewed` | Пользователь увидел карточку | `{recId}` |
| `recommendation.status_changed` | Смена статуса | `{recId, from, to}` |
| `recommendation.text_copied` | Скопирован текст | `{recId}` |
| `recommendation.feedback` | Оставлена обратная связь | `{recId, rating, hasComment}` |

**AI-сервис:**

| eventType | Когда | payload |
|-----------|-------|---------|
| `ai.request` | Отправлен запрос к AI | `{promptKey, model, inputTokens}` |
| `ai.response` | Получен ответ | `{promptKey, outputTokens, durationMs}` |
| `ai.error` | Ошибка AI | `{promptKey, error}` |
| `ai.fallback` | Сработал fallback | `{promptKey, reason}` |

**Навигация:**

| eventType | Когда | payload |
|-----------|-------|---------|
| `page.view` | Открыт экран | `{page: "inbox"|"tasks"|"data"}` |

### 9.3. Хранение и доступ

- Все события хранятся в коллекции `EventLog` в MongoDB.
- Retention: не менее 12 месяцев.
- Доступ: только admin через Payload Admin UI.
- Для MVP достаточно просмотра через админку. В будущем — экспорт в аналитическую систему (PostHog, Amplitude).
