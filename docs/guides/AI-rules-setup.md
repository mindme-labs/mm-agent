## What this is

The product's rules engine has been refactored so that:

- **Deterministic logic** (which counterparties trigger a rule, what amounts and signals) stays in TypeScript — fast, testable, reproducible.
- **Natural-language output** (recommendation title, description, "what to do," ready-to-send letter text) is generated by **per-rule AI prompts** stored in the database.
- **Static templates** remain as a guaranteed failsafe — if AI is disabled, the API key is missing, the call times out, or the response is malformed, the user always sees a usable recommendation.

The system is currently rolled out as a **pilot for rule `ДЗ-1` only** (Просроченная дебиторская задолженность). All other rules continue to run unchanged through the legacy template path.

---

## Architecture at a glance

```
   File upload → parse → metrics → rules engine
                                       │
                                       ▼
                              RuleCandidate[]
                                       │
                  POST /api/analysis/run (fast, <2s)
                                       │
                                       ▼
                        recommendations table
                  (aiEnhanced=false for AI-eligible,
                   aiEnhanced=true for legacy/disabled)
                                       │
                                       ▼
       Client polls POST /api/analysis/ai-enhance-batch
                                       │
                  per batch (default 3 candidates):
                  ├─ load rule_<code> prompt from DB
                  ├─ call Claude with bounded concurrency
                  ├─ on success → update rec, mark aiEnhanced=true
                  └─ on failure → keep fallback text, stamp aiEnhanceFailedAt
                                       │
                                       ▼
                    GET /api/analysis/status
                  (client polls until done: true)
```

## Part 1 — One-time setup

### 1.1 Environment variables

Confirm `.env` (or your Vercel project environment) contains:

```env
ANTHROPIC_API_KEY=sk-ant-...
PAYLOAD_SECRET=<at least 32 chars>
PAYLOAD_PUBLIC_SERVER_URL=https://your-domain.example
MONGODB_URI=mongodb+srv://...
```

Without `ANTHROPIC_API_KEY`, the system runs in 100% fallback mode (everything works, but you see only static template text — no AI enhancement).

### 1.2 Generate Payload types after collection changes

If you pulled the latest code or modified any collection / global settings:

```bash
npm run generate:types
```

This regenerates `src/payload-types.ts` so TypeScript knows about the new `signals`, `aiEnhanceFailedAt`, `aiEnhanceError` fields on `Recommendations` and the new `aiRulesEnabled` / `aiRulesEnabledFor` / `aiRulesBatchSize` fields on `GlobalSettings`.

Sanity check:

```bash
npx tsc --noEmit
```

Should exit 0.

### 1.3 Seed the AI prompts into the database

The `rule_dz1` prompt and the legacy `DEFAULT_PROMPTS` (`audit_working_capital`, `enhance_recommendation`, etc.) live in `src/lib/ai/rule-prompts.ts` and `src/lib/ai/prompts.ts` as code. They must be inserted into the `ai-prompts` MongoDB collection before the system can read them.

**Two ways to seed:**

**Option A — Admin button (recommended)**

1. Open the admin panel: `https://your-domain.example/8ca90f70`
2. Log in as an admin user
3. On the dashboard, click the **"Seed Prompts"** button (custom component above the dashboard)
4. You should see a toast confirming `created: N, skipped: M, total: N+M`

**Option B — Direct API call**

Authenticated as an admin (cookie set by logging into the admin panel):

```bash
curl -X POST https://your-domain.example/api/ai/seed-prompts \
  -H "Cookie: payload-token=<your-admin-cookie>"
```

Re-running is safe — the endpoint skips prompts that already exist (looked up by `promptKey`).

### 1.4 Configure global settings

Open the admin panel → **Globals → Настройки**. Set:

| Field | Value | Why |
|-------|-------|-----|
| `aiEnabled` | ✅ `true` | Master switch for the legacy AI features (`audit_working_capital`, `enhance_recommendation`) |
| `aiProvider` | `anthropic` | Only Anthropic is wired up today |
| `aiModel` | `claude-sonnet-4-6` | Or whichever model your account supports |
| **`aiRulesEnabled`** | ✅ `true` | **Master switch for the new AI-augmented rule pipeline** |
| **`aiRulesEnabledFor`** | `['ДЗ-1']` | Per-rule allowlist. Pilot scope is just `ДЗ-1` |
| **`aiRulesBatchSize`** | `3` (Hobby) / `5` (Pro) | How many candidates the batch endpoint processes per call. Smaller = safer for Vercel timeouts |

Save.

### 1.5 Verify the prompt is loaded

In the admin panel, navigate to **Collections → AI-промпты**. You should see a row with:

- `promptKey`: `rule_dz1`
- `name`: `AI-анализ ДЗ-1 (просроченная дебиторка)`
- `version`: `1`
- `isActive`: ✅ `true`

If absent, re-run seeding (step 1.3).

## Part 2 — How the pipeline runs

### 2.1 The user's perspective

1. User uploads CSV files via the onboarding wizard
2. User clicks "Запустить анализ" (or the demo seed runs)
3. Backend immediately persists recommendations using **fallback text** (so the inbox is non-empty within ~2 seconds)
4. Client begins polling the batch endpoint to enhance recommendations one batch at a time
5. As each batch completes, the inbox cards for `ДЗ-1` items get **replaced in-place** with AI-generated text
6. A small progress indicator shows "AI-анализ: 7 из 12..."
7. When all done, the indicator disappears

### 2.2 The endpoints

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/analysis/run` | Parse files, run rules → candidates, persist recs (fast) |
| `POST` | `/api/analysis/ai-enhance-batch` | Enhance next K candidates with AI |
| `GET`  | `/api/analysis/status` | Get progress: `{ total, enhanced, remaining, failed, done }` |
| `POST` | `/api/ai/seed-prompts` | Admin-only: seed default prompts into DB |
| `GET`  | `/api/ai/status` | Check whether AI is currently available (key + toggle) |

### 2.3 Manually exercising the pipeline

Useful for validation or smoke testing. As an authenticated user:

```bash
# 1. Trigger the rules engine (after files are uploaded or via demo seed)
curl -X POST https://your-domain.example/api/analysis/run \
  -H "Cookie: payload-token=<your-cookie>"
# → { ok: true, total: 12, pendingAi: 5, prefilled: 7, ... }

# 2. Poll the batch endpoint until done
while true; do
  RESULT=$(curl -s -X POST https://your-domain.example/api/analysis/ai-enhance-batch \
    -H "Cookie: payload-token=<your-cookie>" \
    -H "Content-Type: application/json" \
    -d '{"batchSize": 3}')
  echo "$RESULT"
  if echo "$RESULT" | grep -q '"done":true'; then break; fi
  sleep 2
done

# 3. Check final state
curl -s https://your-domain.example/api/analysis/status \
  -H "Cookie: payload-token=<your-cookie>"
# → { phase: 'rules_done', total: 12, enhanced: 12, remaining: 0, failed: 0, done: true }
```

## Part 3 — Prompt catalog: what runs when

The system uses **two separate sets of AI prompts** that serve different purposes. Knowing which prompt is invoked from which code path is critical when authoring or troubleshooting.

### 3.1 The two prompt sources

| Source | File | Loaded by | Purpose |
|--------|------|-----------|---------|
| `DEFAULT_PROMPTS` | `src/lib/ai/prompts.ts` | `/api/ai/seed-prompts` | Legacy AI features and stub prompts authored before the rule pipeline existed |
| `RULE_PROMPTS` | `src/lib/ai/rule-prompts.ts` | `/api/ai/seed-prompts` | Per-rule prompts for the new AI-augmented rules pipeline |

Both sets are seeded into the same `ai-prompts` MongoDB collection. The distinction is purely organizational — at runtime, every prompt is loaded by `loadPrompt(promptKey)` from `src/lib/ai/client.ts`.

### 3.2 Full catalog

| Prompt key | Status | Used by | When it fires | Required? |
|-----------|--------|---------|---------------|-----------|
| **`rule_dz1`** | ✅ **Active** (Phase 1 pilot) | `analyzeCandidates()` in `src/lib/ai/rule-analyzer.ts` | `POST /api/analysis/ai-enhance-batch` for each `ДЗ-1` candidate when `aiRulesEnabled=true` and `ДЗ-1 ∈ aiRulesEnabledFor` | Required for AI rules pilot. Without it, all `ДЗ-1` recs use static template |
| `audit_working_capital` | ✅ Active (legacy) | `runAIAudit()` in `src/lib/ai/audit.ts` | `POST /api/analysis/ai-audit` and `seedDemoForUser()` when `aiEnabled=true` and `ANTHROPIC_API_KEY` set | Optional. Generates 2-3 strategic recommendations alongside rule output |
| `enhance_recommendation` | ✅ Active (legacy, manual) | `POST /api/analysis/ai-enhance` | Manual single-recommendation enhancement (one rec per call, sequentially). Was the previous "improve text" path before the rule pipeline existed | Optional. **Not used by the new rule pipeline** — superseded by `rule_<code>` prompts |
| `recommendation_text` | ⚠️ Stub | None today | Authored for general letter-generation use; no current code path invokes it | Skip if not editing |
| `file_recognition` | ⚠️ Stub (planned) | None today | Designed for AI fallback when `parseOSVFile()` regex fails on a non-canonical CSV header. Will be wired in a future "AI file extraction" sprint (separate from this initiative) | Skip until file-extraction sprint |
| `data_extraction` | ⚠️ Stub (planned) | None today | Designed for AI extraction of full structured data from non-canonical OSV files. Same future sprint as above | Skip until file-extraction sprint |

### 3.3 Decision matrix — which prompts to seed

| Your goal | Must-have prompts | Optional prompts |
|-----------|-------------------|------------------|
| **Pilot AI rules pipeline (Phase 1)** | `rule_dz1` | `audit_working_capital` (if you want the strategic audit), `enhance_recommendation` (if you keep the manual single-rec path active) |
| **Run product without AI rules** (rules engine only) | None | `audit_working_capital` if you want AI strategic audit on top of static rules |
| **Full Phase 2 rollout** (all 9 rules AI-enhanced) | `rule_dz1` + the 8 not-yet-authored: `rule_dz2`, `rule_dz3`, `rule_kz1`, `rule_zap1`, `rule_zap2`, `rule_pl1`, `rule_fc1`, `rule_svs1` | `audit_working_capital` |
| **Disable all AI** | — | — |

The `POST /api/ai/seed-prompts` button always seeds **everything** — both `DEFAULT_PROMPTS` and `RULE_PROMPTS`. Skipping happens automatically by `promptKey` uniqueness, so seeding is idempotent and safe to repeat.

### 3.4 Prompt invocation flow per scenario

#### Scenario A — User runs analysis with AI rules enabled (current state, pilot)

```
POST /api/analysis/run
   │
   ├─ runRulesEngine() produces 12 candidates
   │   ├─ 5 candidates from runDZ1 (native RuleCandidate[])
   │   └─ 7 candidates from legacy rules (wrapped via legacyToCandidate)
   │
   ├─ persists 12 recommendations:
   │   ├─ 5 ДЗ-1 → aiEnhanced=false (will be enhanced by batch)
   │   └─ 7 legacy → aiEnhanced=true (already final via templates)
   │
   └─ returns { total: 12, pendingAi: 5, prefilled: 7 }

Client polls POST /api/analysis/ai-enhance-batch (3 candidates per call)
   │
   └─ for each ДЗ-1 candidate:
       └─ analyzeOne() → callAI({ promptKey: 'rule_dz1', ... })
           └─ Anthropic → JSON response → mergeAIResponse() → DB update
```

Prompts invoked: **`rule_dz1`** only. Other rules don't trigger any AI call in this path.

#### Scenario B — Demo seeding for a new user

```
POST /api/demo/seed → seedDemoForUser()
   │
   ├─ runRulesEngine() → candidates (same as Scenario A)
   │
   ├─ if isAIAvailable():
   │   └─ runAIAudit() → callAI({ promptKey: 'audit_working_capital' })
   │       └─ produces 2-3 AI-AUDIT recommendations (separate from rule recs)
   │
   ├─ persists rule-based recommendations (ДЗ-1 with aiEnhanced=false)
   ├─ persists AI-AUDIT recommendations (aiEnhanced=true, isAiGenerated=true)
   └─ returns total count
```

Prompts invoked: **`audit_working_capital`** synchronously during seed; **`rule_dz1`** later when client polls the batch endpoint.

#### Scenario C — User clicks "Improve with AI" on a single recommendation (legacy manual path)

```
POST /api/analysis/ai-enhance
   │
   └─ picks 1 unenhanced recommendation (any rule)
       └─ callAI({ promptKey: 'enhance_recommendation', ... })
```

Prompts invoked: **`enhance_recommendation`**. This path is independent of the new rule pipeline. If a `ДЗ-1` rec has already been enhanced via `rule_dz1`, this endpoint will skip it (filter: `aiEnhanced != true`).

#### Scenario D — AI completely disabled

If `aiRulesEnabled=false` AND `aiEnabled=false`:

```
POST /api/analysis/run → all candidates persisted with template text, aiEnhanced=true
POST /api/analysis/ai-enhance-batch → returns { done: true, processed: 0 } immediately
POST /api/analysis/ai-audit → returns 503 (AI unavailable)
```

No prompts invoked. Inbox shows static template text for everything. Fully functional product.

- `enhance_recommendation` → no field updates, raw text saved as `fullText`

### 3.7 What if I add a brand-new rule (Phase 2)?

You need both code changes AND a new prompt. The naming convention is strict: the analyzer expects `rule_<lowercased rulecode with non-alphanumeric stripped>` (see `promptKeyForRule()` in `src/lib/ai/rule-prompts.ts`).

| Rule code | Expected `promptKey` |
|-----------|---------------------|
| `ДЗ-1` | `rule_dz1` |
| `ДЗ-2` | `rule_dz2` |
| `ДЗ-3` | `rule_dz3` |
| `КЗ-1` | `rule_kz1` |
| `ЗАП-1` | `rule_zap1` |
| `ЗАП-2` | `rule_zap2` |
| `ПЛ-1` | `rule_pl1` |
| `ФЦ-1` | `rule_fc1` |
| `СВС-1` | `rule_svs1` |
| `AI-AUDIT` | n/a — uses `audit_working_capital` instead |

If `aiRulesEnabledFor` lists a rule but the matching `rule_<code>` prompt is missing in the database, every candidate for that rule will fail with `aiEnhanceError: 'ai_invalid_json'` (because `loadPrompt()` returns null and `callAI` returns null) and fall back to the static template. Always seed prompts before adding the rule code to `aiRulesEnabledFor`.

## Part 4 — Authoring & tuning prompts

### 4.1 Prompt file structure

A prompt is a row in the `ai-prompts` collection with these fields:

| Field | Purpose |
|-------|---------|
| `promptKey` | Unique key. Must match `rule_<lowercased rule code>` for the analyzer to find it |
| `name` | Human-readable label for the admin UI |
| `systemPrompt` | The system message sent to Claude. Defines persona, tone, output format |
| `userPromptTemplate` | The user message template with `{variable}` placeholders. Variables come from `RuleCandidate.signals` plus company metrics |
| `version` | Integer. Analyzer always uses the highest active version per `promptKey` |
| `isActive` | Toggle to deactivate without deleting |

### 4.2 Variables available in `userPromptTemplate`

The analyzer (`src/lib/ai/rule-analyzer.ts → buildPromptVariables`) injects:

**Always present:**
- `{ruleCode}`, `{ruleName}`, `{priorityHint}`
- `{counterparty}`, `{recipient}`
- `{impactAmount}`, `{sourceAccount}`

**Company context (from latest analysis):**
- `{revenue}`, `{grossMargin}`
- `{accountsReceivable}`, `{accountsPayable}`
- `{arDays}`, `{apDays}`

**Per-rule, from `RuleCandidate.signals`:**

For `rule_dz1`, the rule sets these in `src/lib/rules/dz1-overdue-receivable.ts`:
- `{balance}` — debt amount in rubles
- `{consecutiveNoPayment}` — number of months with zero payments
- `{recentPayments}` — sum of payments in last 2 months
- `{paymentRatio}` — `recentPayments / balance × 100` (integer percent)
- `{penaltyAmount}` — calculated penalty per Russian Civil Code (0.1%/day)
- `{penaltyDays}` — penalty period in days
- `{scenario}` — one of `slowing_payments` / `soft_overdue` / `hard_overdue`

If you reference a variable that wasn't provided, the literal `{variableName}` will leak into the prompt — useful as a debug signal.

### 4.3 Required output JSON shape

Claude must return:

```json
{
  "priority": "critical" | "high" | "medium" | "low",
  "title": "string",
  "description": "string",
  "shortRecommendation": "string",
  "fullText": "string"
}
```

The analyzer:

1. Strips text outside the first `{...}` block (so brief prose around the JSON is tolerated)
2. Validates `title` and `description` are present and strings — otherwise treats as failure
3. Caps `priority` at the rule's hint + 1 level (so AI can't promote a `medium` rule to `critical`)
4. Falls back to the rule's static template on any validation failure

### 4.4 Editing a prompt

**Without releasing code:**

1. Admin panel → **AI-промпты**
2. Open the `rule_dz1` row
3. Edit `systemPrompt` or `userPromptTemplate`
4. Bump `version` (e.g., `1` → `2`)
5. Save

The next batch endpoint call picks up the new version (`sort: '-version'` in `loadPrompt()`).

**To roll back:**

Set `isActive: false` on the new version. The analyzer falls back to the next-highest active version, or to the static template if none.

**To create a new variant for A/B comparison:**

Duplicate the row, change `promptKey` to `rule_dz1_v2`, then... currently the analyzer always uses `rule_<rulecode>` so true A/B requires code changes. For now, the `version` field gives you sequential rollout — author v2, validate against demo data, then deactivate v1.

### 4.5 Testing a new prompt locally

Fastest dev loop:

```bash
# 1. Edit the prompt in admin panel (or in src/lib/ai/rule-prompts.ts then re-seed)
# 2. As a user, reset and re-seed demo
curl -X POST http://localhost:3000/api/dev/reset-onboarding -H "Cookie: payload-token=..."
curl -X POST http://localhost:3000/api/demo/seed -H "Cookie: payload-token=..."
# 3. Trigger analysis + enhancement
curl -X POST http://localhost:3000/api/analysis/run -H "Cookie: payload-token=..."
curl -X POST http://localhost:3000/api/analysis/ai-enhance-batch -H "Cookie: payload-token=..."
# 4. Inspect results in /app/inbox or via admin panel → Recommendations
```

## Part 5 — Operations

### 5.1 Monitoring AI cost

Every AI call is logged to `ai-usage-logs` with token counts and computed cost.

Admin panel → **Логи AI-запросов** shows:

| Column | Meaning |
|--------|---------|
| `promptKey` | Filter by `rule_dz1` to see only AI-rule calls |
| `inputTokens`, `outputTokens` | Token usage |
| `cost` | USD, computed at $3/M input + $15/M output |
| `durationMs` | Latency per call |

Quick aggregations via the Payload REST API:

```bash
curl "https://your-domain.example/api/ai-usage-logs?where[promptKey][equals]=rule_dz1&limit=100" \
  -H "Cookie: payload-token=<admin-cookie>"
```

### 5.2 Monitoring quality / failures

The `event-log` collection records each AI lifecycle event (`ai.request`, `ai.response`, `ai.error`, `ai.fallback`). Filter by `eventType = 'ai.error'` to see failures.

For per-recommendation forensics, the `recommendations` collection now has:

- `aiEnhanced` (boolean) — true once successfully enhanced
- `aiEnhanceFailedAt` (date) — last failure timestamp
- `aiEnhanceError` (string) — short error code: `ai_timeout_or_unavailable`, `ai_invalid_json`, or the underlying exception message

Recommendations with `aiEnhanced=false AND aiEnhanceFailedAt set` are stuck. They retry automatically after a 5-minute cooldown when the batch endpoint is called again.

### 5.3 Forcing a retry of failed enhancements

Just call the batch endpoint again — failed recs become eligible 5 minutes after their last failure stamp. Or to retry immediately, manually clear `aiEnhanceFailedAt` in the admin panel for the affected recommendations.

### 5.4 Disabling AI in an emergency

Two options, in order of severity:

**Stop AI for all rules (instant, recoverable):**

Admin panel → **Настройки** → uncheck `aiRulesEnabled` → save.

Effect: next analysis run persists all candidates with `aiEnhanced=true` using fallback text. The batch endpoint returns `done: true` immediately. Existing recommendations are unchanged.

**Disable AI for one rule only:**

Admin panel → **Настройки** → remove the rule from `aiRulesEnabledFor` → save.

**Hard kill — disable AI globally:**

Remove `ANTHROPIC_API_KEY` from environment and redeploy. The `isAIAvailable()` check returns false, every AI call returns null, fallback path activates everywhere.

### 5.5 Cost guardrails

Soft guardrails in place today:

- `aiRulesBatchSize` caps how many candidates run per HTTP call (limits damage if a runaway loop occurs)
- 5-minute retry cooldown prevents thrashing on persistent failures
- Per-call timeout (15 s) caps individual call cost

Hard guardrails are **not yet implemented** — Phase 3 of the dev plan adds:
- Per-analysis cost cap that aborts remaining enhancements
- Daily quota enforcement in `callAI`
- `analysis-results.aiCost` for per-analysis cost tracking

Until then, monitor `ai-usage-logs` daily and alert manually if you see runaway usage.

## Part 6 — Common scenarios & troubleshooting

### Scenario: I enabled `aiRulesEnabled` but recommendations still show static text

Checklist:

1. Is `ДЗ-1` in `aiRulesEnabledFor`? (admin panel → Настройки)
2. Is the `rule_dz1` row present and `isActive: true` in `ai-prompts`? (admin panel → AI-промпты)
3. Is `ANTHROPIC_API_KEY` set? Hit `GET /api/ai/status` → expect `{ available: true }`
4. Did you actually call `/api/analysis/ai-enhance-batch`? `/api/analysis/run` only persists candidates — it does NOT enhance them
5. Are the recommendations actually `ДЗ-1`? Other rules use legacy templates — their `aiEnhanced` is set to `true` immediately by `/api/analysis/run` to skip the batch endpoint, but they were never AI-enhanced in this release
6. Check `aiEnhanceError` on the affected recommendation in admin panel → Recommendations

### Scenario: Batch endpoint times out on Vercel

Symptoms: 504 from `/api/analysis/ai-enhance-batch` after ~10 s.

Fixes:

1. Reduce `aiRulesBatchSize` to `2` (admin panel → Настройки)
2. If on Hobby, upgrade to Pro for `60 s` default timeout
3. Check Anthropic API status — if their latency spiked, your batch will too
4. As a hotfix, set `aiRulesEnabled: false` to bypass AI entirely

### Scenario: AI returns wrong/garbled output

1. The analyzer rejects malformed JSON automatically and falls back. Check `aiEnhanceError = 'ai_invalid_json'` for affected recs
2. If accepted but quality is poor, check the prompt: open `rule_dz1` in admin panel
3. Review the actual response in `event-log` (`eventType = 'ai.response'`) and `ai-usage-logs`
4. Iterate on `systemPrompt` — common improvements: stricter JSON schema in the prompt, explicit "Don't use Markdown", more concrete tone instructions, examples of good output
5. Bump `version` and save — next batch picks up the new prompt

### Scenario: A user's analysis has `failed > 0` after polling

This means some recommendations could not be AI-enhanced and are showing fallback text. The user can still act on them — fallback text is fully usable. To force re-attempt:

1. Wait 5 minutes for the cooldown
2. Have the client poll `/api/analysis/ai-enhance-batch` again
3. Or manually clear `aiEnhanceFailedAt` in admin panel

### Scenario: I want to add an AI prompt for `КЗ-1` (or any other rule)

This is Phase 2 work — not yet shipped. The minimum changes required:

1. Refactor `src/lib/rules/kz1-unclosed-advances.ts` to return `RuleCandidate[]` (mirror what `dz1-overdue-receivable.ts` does)
2. Add `fallbackForKZ1` to `src/lib/rules/fallback-templates.ts` and register it
3. Remove `runKZ1` from `LEGACY_RULES` in `src/lib/rules/engine.ts` and call it directly via the candidates path (mirror `runDZ1`)
4. Add a new entry to `RULE_PROMPTS` in `src/lib/ai/rule-prompts.ts` with `promptKey: 'rule_kz1'`
5. Re-seed prompts (admin button)
6. Add `КЗ-1` to `aiRulesEnabledFor` in admin panel

## Part 8 — When to escalate

Contact the platform team if you see:

- Sustained `aiEnhanceError` rate >10% across users → likely a prompt or model issue
- Daily AI cost trending above your monthly budget / 30 → enable hard cap (Phase 3)
- Wall time per batch routinely >8s on Pro → batch size or Anthropic latency issue
- Users reporting the inbox is empty for >30s after analysis starts → `/api/analysis/run` failing or extremely slow file parsing
